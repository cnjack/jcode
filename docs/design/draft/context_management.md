# Context Management: Summarization & Tool Output Reduction

## 1. Problem Statement

Long conversations and large tool outputs cause context-window overflow, which terminates the agent run with a hard API error. There is currently no mechanism to compress history or truncate oversized tool results.

Two concrete failure scenarios:

**Scenario A — History Overflow**: A coding session with many back-and-forth exchanges, multiple file reads, and large execute outputs gradually accumulates tokens. When the total exceeds the model's context window (e.g. 128k tokens for GPT-4o), the next `Generate()` call fails with a context-length error and the entire session is lost.

**Scenario B — Single Tool Explosion**: The user asks the agent to read a 5000-line file. The `read` tool returns ~200k characters. Even before context accumulates, this single result blows the limit.

Without any mitigation, both are permanent failures requiring a full restart.

## 2. Goals & Non-Goals

**Goals**
- Automatically compress conversation history when the token count approaches the context window limit (summarization).
- Truncate and offload oversized individual tool outputs (reduction).
- Make both thresholds configurable with safe defaults.

**Non-Goals**
- Does not change tool interfaces or add new tools.
- Does not implement a custom summarization prompt — relies on Eino's built-in middleware.
- Does not change session JSONL format (the compressed history is internal to the agent run, not persisted).

## 3. Current State

`internal/agent/agent.go` creates the agent with no `Handlers`:

```go
return adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    // ...
    Middlewares: middlewares,
    // Handlers field is absent
})
```

There is no protection against context-window exhaustion. The `MaxIterations: 1000` cap may never be reached because the model errors out first.

Token usage is tracked globally in `internal/model/chatmodel.go` via `TokenTracker`, but this value is only displayed in the TUI — the agent itself has no awareness of how close it is to the limit.

## 4. Proposed Design

### 4.1 Eino Built-in Middleware

Both mitigations use Eino's built-in ADK middlewares, which implement `ChatModelAgentMiddleware` and slot into the `Handlers` field of `ChatModelAgentConfig`.

```go
import (
    "github.com/cloudwego/eino/adk/middlewares/summarization"
    "github.com/cloudwego/eino/adk/middlewares/reduction"
)
```

### 4.2 Summarization Middleware

When the total tokens in the conversation state exceed a threshold, the middleware replaces the accumulated message history with a compressed summary generated by the same model, preserving the essential context.

```go
summarizationMW, err := summarization.New(ctx, &summarization.Config{
    Model: chatModel,
    Trigger: &summarization.TriggerCondition{
        ContextTokens: summarizationThreshold(contextLimit),
    },
})
```

The threshold function chooses 75% of the model's context limit as the default trigger:

```go
func summarizationThreshold(contextLimit int) int {
    if contextLimit <= 0 {
        return 100_000 // safe absolute default
    }
    return int(float64(contextLimit) * 0.75)
}
```

Context limit is read from `internalmodel.GetModelContextLimit(cfg.Model)` (already implemented).

### 4.3 Tool Output Reduction Middleware

Prevents a single tool result from exceeding a per-output character budget. Content beyond the limit is offloaded to a local file, and the agent receives a truncated result with a pointer to the full content.

```go
reductionMW, err := reduction.New(ctx, &reduction.Config{
    Backend:           reductionBackend(), // ~/.jcoding/reduction/
    MaxLengthForTrunc: 50_000,             // chars per single tool output
    MaxTokensForClear: 30_000,             // total tool-output tokens before clearing
})
```

`reductionBackend()` creates a filesystem backend rooted at `~/.jcoding/reduction/`. This directory persists across sessions, allowing the agent to reference offloaded content if needed.

### 4.4 New Config Fields

```json
// ~/.jcoding/config.json
{
  "SummarizationThreshold": 0,
  "MaxToolOutputChars": 50000,
  "MaxToolOutputTokens": 30000
}
```

All default to zero, meaning use the code-level defaults above.

```go
// internal/config/config.go
type Config struct {
    // existing fields ...
    SummarizationThreshold int `json:"SummarizationThreshold,omitempty"`
    MaxToolOutputChars      int `json:"MaxToolOutputChars,omitempty"`
    MaxToolOutputTokens     int `json:"MaxToolOutputTokens,omitempty"`
}
```

### 4.5 Integration in `agent.go`

Middlewares are passed via the `Handlers` field, which is separate from `Middlewares` in `ChatModelAgentConfig`:

```go
return adk.NewChatModelAgent(ctx, &adk.ChatModelAgentConfig{
    // ...
    Handlers: []adk.ChatModelAgentMiddleware{
        summarizationMW,       // compresses history at 75% token usage
        reductionMW,           // truncates individual oversized tool outputs
        &safeToolMiddleware{}, // innermost: error handling (see middleware_pipeline design)
    },
    Middlewares: middlewares, // approval, telemetry
})
```

### 4.6 Token Budget Awareness (Coordination with System Reminders)

The System Reminders design (separate document) injects token-usage warnings at 60% and 85%. This creates a cooperative layered protection:

```
Token usage:
0% ──────── 60% ────── 75% ──────── 85% ──────── 100%
            │           │             │
     Reminder:       Summarization  Reminder:
   "watch length"   (auto compact)  "finish soon"
```

The reminders inform the agent to produce shorter outputs *before* compaction fires, reducing the frequency of summarization calls.

## 5. Alternatives Considered

### Implement summarization manually in `runner.go`
Rejected. Eino's built-in middleware handles token counting, uses the model to generate the summary, and integrates with the agent state management. A manual implementation would duplicate this complexity and create a maintenance burden.

### Use `/tmp` instead of `~/.jcoding/reduction/` for offloaded tool output
`~/.jcoding/reduction/` is preferred because: (1) offloaded content survives session boundaries and can be referenced across multiple turns; (2) the user can inspect oversized outputs for debugging. `/tmp` is cleaned automatically and is unsuitable for content referenced in the conversation history. A TTL-based cleanup of `~/.jcoding/reduction/` can be added later.

### Fixed 100k-character limit regardless of model
Rejected. Different models have different context windows. Tying the threshold to `GetModelContextLimit()` ensures the behaviour adapts when the user switches models via the TUI.

## 6. Migration & Compatibility

- Only `internal/agent/agent.go` and `internal/config/config.go` change.
- The new config fields are optional with zero-value fallbacks.
- The `~/.jcoding/reduction/` directory is created on first use if absent.
- No change to session JSONL, TUI messages, or tool interfaces.

## 7. Open Questions

1. **Eino version**: Do `github.com/cloudwego/eino/adk/middlewares/summarization` and `reduction` exist in the version pinned in `go.mod`? If not, what is the minimum version required?
2. **Summarization cost**: The summarization middleware uses the same model to generate summaries. For very expensive models, this is an unexpected background cost. Should there be a flag to disable summarization entirely (`SummarizationThreshold: -1`)?
3. **Reduction backend directory**: Should `~/.jcoding/reduction/` be cleaned up automatically (e.g. files older than 7 days)? Or left to the user?
